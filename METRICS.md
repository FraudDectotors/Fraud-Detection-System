# Implementations Of Software Metrics

## 1. Measurement Theory

Measurement theory ensures that collected metrics are well-defined and meaningful. In this project, measurements are obtained when each fraud detection algorithm analyzes the same `Transaction` dataset and returns a `DetectionResult`.

### Entities Being Measured
- Execution time (`DetectionResult.timeMs`)
- Number of suspicious transactions flagged (`DetectionResult.fraudFound`)
- List of flagged transactions (`DetectionResult.flagged`)

### Measurement Scale Type
This fraud detection system uses nominal and ratio scales.

| Entity | Scale Type | Justification |
|------|------|------|
| Algorithm name | Nominal | Label used to classify algorithms; no arithmetic operations are meaningful. |
| Execution time | Ratio | Measured in milliseconds, has a true zero, and supports meaningful ratios. |
| Number of flagged transactions | Ratio | A count with true zero. |

### Valid Operations And Summaries
Because `timeMs` and `fraudFound` are ratio measures, these operations are valid:
- Mean (average) across runs
- Min and max across runs
- Standard deviation across runs (stability/variation)
- Ratio comparison, for example Algorithm A is 1.2x slower than Algorithm B

### Measurement Quality Considerations (Validity And Reliability)
Reliability (repeatability) can be affected by runtime noise:
- JVM warm-up and JIT compilation
- OS scheduling and background processes

This is why the system supports repeated runs (`numRuns`) and compares measurements statistically.

### Are We Measuring What We Claim?
`fraudFound` is not true fraud confirmed by labels. It is the number of transactions flagged as suspicious by an algorithm.
The generated dataset has no labeled ground truth, so the project should not claim true detection accuracy metrics such as precision or recall.

## 2. Goal-Based Software Measurement Framework (GQM Style)

A goal-based framework links why we measure to what we measure.

### Goal (G)
Evaluate and compare fraud detection algorithms (rule-based vs statistical) in terms of efficiency and flagging behavior on the same synthetic dataset.

- Object: fraud detection algorithms (`SimpleRuleAlgorithm`, `StatisticalAlgorithm`)
- Purpose: evaluate and compare
- Quality focus: runtime performance and flagging behavior
- Context: synthetic transactions generated by `TransactionGenerator`, thresholds configured in `Main`

### Questions (Q)
1. Which algorithm runs faster on the same dataset?
2. Which algorithm flags more transactions under current thresholds?
3. How consistent are results across repeated runs?

### Metrics (M)
- M1: Execution time (ms), source `DetectionResult.timeMs`
- M2: Number of flagged transactions, source `DetectionResult.fraudFound`

### Mapping (G -> Q -> M)
| Goal | Question | Metric(s) | Where In Code |
|------|------|------|------|
| Compare efficiency | Which algorithm is faster? | `timeMs` | `DetectionResult.timeMs`, reported by `ConsoleReporter` |
| Compare behavior | Which flags more? | `fraudFound` | `DetectionResult.fraudFound`, reported by `ConsoleReporter` |

### Parameters Used
Configured in `main/Main.java`:
- Rule-based parameters:
  - `highAmountThreshold` (default `1_000_000.0`)
  - `foreignLocation` (default `Foreign`)
- Statistical parameter:
  - `zThreshold` (default `0.5`)

These parameters define measurement context and must be reported alongside results.

## 3. Empirical Investigation

Empirical investigation means collecting evidence through controlled experiments, then summarizing and interpreting it.

### Experimental Design
The same dataset is used for each algorithm, so observed differences are more likely due to algorithm behavior than dataset variation.

Independent variables:
- Algorithm (`Rule` vs `Stat`)
- Parameter thresholds (`zThreshold`, `highAmountThreshold`)

Dependent variables:
- Execution time (`timeMs`)
- Flagged count (`fraudFound`)

Controlled variables:
- Same transaction list (`data`) passed to both algorithms
- Same parameter map passed into each algorithm call

### Procedure
Implemented in `experiment/Experiment.java`:
1. Generate synthetic dataset: `TransactionGenerator.generate(dataSize)`
2. For each run from `1..numRuns`, for each algorithm:
   - Run `analyze(data, params)`
   - Collect `DetectionResult`
3. Print per-run comparison in `ConsoleReporter`

### Example Hypotheses
- H1 Efficiency: Rule-based algorithm (`Rule`) has lower mean execution time than statistical (`Stat`) under default parameters.
- H2 Flagging behavior: Statistical algorithm (`Stat`) may flag more transactions at low `zThreshold` values (for example `0.5`).

### Data Analysis Outputs
Per algorithm over repeated runs:
- Mean time (ms)
- Standard deviation of time (ms)
- Mean flagged count
- Standard deviation of flagged count

This supports conclusions such as: Algorithm A is faster on average and more stable (lower mean and lower standard deviation).

- The project now has a moderate codebase size (over 1.7k LOC), so previous "minimalist" size conclusions are no longer valid.
- Comparative fraud algorithm benchmarking is still valid, but numeric outcomes should be reported with date, run count, data size, and parameter values.
