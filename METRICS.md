# IMPLEMENTATIONS OF SOFTWARE METRICS

# 1.MEASUREMENT THEORY

Measurement theory ensures that the metrics we collect are well defined and meaningful. In this project, measurements are obtained when each fraud detection algorithm analyzes the same set of `Transaction` data and returns a `DetectionResult`.

### Entities being measured.
 Execution time. (`DetectionResult.timeMs`)
 Number of suspicious/fraud transactions flagged (`DetectionResult.fraudFound`)
 The list of flagged transactions (`DetectionResult.flagged`)

### Measurement Scale Type.
In this Fraud detection system, we used the nominal and ratio scale.
Entity| Scale type| Justification
------:|:---------:|:-----------
Algorithm name| Nominal|It is label used to classify the algorithm but no arithmetic can be done on it.
Execution time| Ratio| Measured in milliseconds , has a true zero and ratios are meaningful. 
Number of fraud transactions flagged| Ratio| It is a count. 

### Valid operations and summaries

Because `timeMs` and `fraudFound` are ratio measures, the following operations are valid and meaningful:
 Mean (average) across runs
 Min / max across runs
 Standard deviation across runs (helps quantify stability/variation)
 Comparing ratios for example, Algorithm A is 1.2× slower than Algorithm B)

### Measurement quality considerations (validity & reliability)

 Reliability issues (repeatability)
Execution time measured with `System.nanoTime()` can vary due to:
JVM warm-up / JIT compilation
OS scheduling and background processes
  This is why the system supports running multiple runs (`numRuns`) and comparing repeated measurements.

### Are we measuring what we claim?
`fraudFound` is not “true fraud detected.” It is transactions flagged as suspicious by the algorithm.
 The generated dataset does not contain labeled fraud/legitimate ground truth, so we cannot claim true detection accuracy.


# 2. GOAL BASED SOFTWARE MEASUREMENT FRAMEWORK (GQM-style)

A goal-based framework connects why we measure to what we measure.

### Goal (G)
 
Evaluate and compare fraud detection algorithms (Rule-based vs Statistical) in terms of efficiency and flagging behavior when analyzing the same synthetic transaction dataset.

Object: fraud detection algorithms (`SimpleRuleAlgorithm`, `StatisticalAlgorithm`)
Purpose: evaluate/compare
Quality focus: runtime performance and flagging behavior
Context: synthetic transactions generated by `TransactionGenerator` with thresholds configured in `Main`

 ### Questions (Q)


1.  Which algorithm runs faster on the same dataset?
2.  Which algorithm flags more transactions under the current thresholds?
3.  How consistent are the results across repeated runs?

### Metrics (M)

Metrics already present (directly or via `Metric<T>`):

 **M1:** Execution Time (ms)  
Source: `DetectionResult.timeMs`

 **M2:** Number of flagged transactions  
Source: `DetectionResult.fraudFound`


 ### Mapping (G → Q → M)

| Goal | Question | Metric(s) | Where in code |
|------|----------|-----------|---------------|
| Compare efficiency | Which algorithm is faster? | `timeMs` | `DetectionResult.timeMs`, printed by `ConsoleReporter` |
| Compare behavior | Which flags more? | `fraudFound` | `DetectionResult.fraudFound`, printed by `ConsoleReporter` |


 ### Parameters  used

Configured in `main/Main.java`:

 Rule-based parameters:
   `highAmountThreshold` (default 1,000,000.0)
   `foreignLocation` (default "Foreign")
 Statistical parameter:
   `zThreshold` (default 0.5)

These parameters define the context of the measurement and are reported alongside results to keep measurements interpretable.



#  3. EMPIRICAL INVESTIGATION.

Empirical investigation means systematically collecting evidence through controlled experiments, then summarizing and interpreting it.

### Experimental design in this project
  
 The same dataset is used for each algorithm, so differences are more likely due to the algorithm rather than dataset differences.

**Independent variable(s)**
 Algorithm (`Rule` vs `Stat`)
 Parameter thresholds (e.g., `zThreshold`, `highAmountThreshold`)

**Dependent variable(s)**
 Execution time (`timeMs`)
 Flagged count (`fraudFound`)

**Controlled variables**
 Same list of transactions `data` passed to both algorithms.
 Same parameter map passed into each algorithm call.

###  Procedure (how the experiment runs)

Implemented in `experiment/Experiment.java`:

i. Generate a synthetic dataset: `TransactionGenerator.generate(dataSize)`
ii. For run = 1..numRuns:
    For each algorithm:
      Run `analyze(data, params)`
      Collect `DetectionResult`
iii. Print results per run in `ConsoleReporter`

### Hypotheses 

These are example hypotheses you can state before running:

 **H1 Efficiency:** The rule-based algorithm (`Rule`) will have lower mean execution time than the statistical algorithm (`Stat`) because it performs a single pass with simple checks.
 
 **H2 Flagging behavior:** The statistical algorithm (`Stat`) will flag more transactions than the rule-based algorithm (`Rule`) under the chosen `zThreshold` (0.5), because it treats high values as outliers more broadly.

### Data collection and analysis 
A summary statistics over all runs is output for each algorithm.

 Mean time (ms)
 Standard deviation of time (ms)
 Mean flagged count
 Standard deviation of flagged count


This allows conclusions such as:
“Algorithm A is faster on average and more stable” (lower mean and lower stddev).
